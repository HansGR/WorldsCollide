# Programmer's Guide: Door Randomization (`doors/`)

The `doors/` package implements every door-randomization mode: `-dre`,
`-drdc`, `-dra`, `-drx`, `-maps`/`-mapx`, the individual `-dr*` flags,
and `-ruin`. This guide is for working *inside* the package; development
history and milestones are in doors/HISTORY.md.

## 1. The two sentences that matter

**Everything is planned once, in the Data phase, as pure Python over static
data; the ROM is only touched afterward, by binding/realization code that
reads the finished plan.** If you find yourself wanting to read the ROM, an
`args` global, or a live event object inside `doors/`, you are in the wrong
layer â€” pass the fact in as data.

## 2. Layers (each only imports downward)

```
doors/
  atlas/           WHAT EXISTS   generated exit truth (never hand-edit compiled.py)
  model.py         WHAT IS CONNECTED   journaled world state
  plan/            WHAT TO CONNECT     pure RNG-consuming planners
  validate/        IS IT RIGHT         structural acceptance gates
  realize/         HOW IT'S WRITTEN    door_map build, exit connection,
                   one-way transitions, event-tile runtime updates
                   (functions take the live Maps object; import-clean)
```

- **`atlas/compiled.py`** is GENERATED by `tools/compile_atlas.py` from
  `claude_reference/exits_raw.json` + the hand-owned tags in
  `atlas/curation.py`. To change partner data: edit curation, run
  `python3 tools/compile_atlas.py` (regenerates), then `--check` (the
  consistency gate: partner parity, reciprocity, room coverage, pool
  solvability). Stale curation entries are detected and refused.
- **`atlas/room_names.py`** â€” the human-legible room registry
  (`ZOr01`-style codes); `doors.atlas.room_name(id)` resolves them.

## 3. The model (`doors/model.py`)

`WorldModel(rooms)` holds the planner's whole mutable state:

- Rooms merge into **clusters** (mutually-reachable sets) via a rollback
  union-find; one-way trapâ†’pit edges form a cluster graph that is kept a
  **DAG at all times** (closing a cycle merges every cluster on it).
- Keys/locks: `apply_key` opens every lock whose
  key tuple is held; released exits go live *and* into
  `initially_locked_exits` (usable as sources, never as walk targets).
- **Every mutation is journaled.** Backtracking is
  `mark = w.checkpoint()` â€¦ `w.rollback(mark)` â€” no deepcopy anywhere.
- `add_room` (dynamic pool growth for ruination) assesses the new room's
  locks against the current keychain â€” the Mogâ†’`lw1`â†’Lone Wolf rule.
- Two id caveats confined here: `_element_kind` classifies *locked* ids by
  range; `live_kind` classifies *live* elements by list membership and is
  authoritative where ranges lie (door-as-trap exits like 182 in room 61).

## 4. Planning

- **`plan/walk.py`** â€” the backtracking walk: offer exits from the
  active cluster + its downstream,
  try random legal entrances, prune with `plan/prune.py` Rules Aâ€“F,
  rollback on failure, deterministic attempt budget. Protection filters
  **targets only**; forced exits fail fast to their forced entrance.
- **`plan/modes.py`** â€” `plan_for_args(args, rng, characters)` is the one
  dispatch for every mode. Pools come from `data/room_sets.py`; shared-exit
  splits, protect-door replacement (30000+), virtual roots (10000+), and
  crossworld links (20000/20001) are **spec transforms on fresh copies** â€”
  no shared table is ever mutated.
- **`plan/artifact.py`** â€” every mode returns the same `DoorPlan`
  (`door_pairs`, `oneways`, `ruination: RuinPlan | None`). `Doors.mod`
  stores it on `doors.plan`; Events receives it.
- **`plan/ruination/`** â€” see the package docstring for the module map.
  The shape in one paragraph: three `RuinBranch` views share ONE
  `WorldModel` (element uniqueness across branches is automatic; the
  keychain is global: a recruit on one branch unlocks all). `growth.py` grows
  branches toward reward rooms, banking character-locked checks and
  cascading recruits (key â†’ areas â†’ banked rewards); `finalize.py` closes
  each branch with the six-step procedure and runs the BFS softlock
  verifier; `RuinConfig` holds per-plan COPIES of every table (`-maze` /
  `-open` adjustments applied at construction), so retries need no reset
  or rollback of anything.

## 7. Recipes

- **Add/repair a door partnership** â†’ `doors/atlas/curation.py` (+ tag),
  regenerate, `--check`.
- **Add a randomizable area** â†’ add its room list to `data/room_sets.py`,
  wire the flag in `plan/modes.py` (`INDIVIDUAL_FLAGS` table), done â€” the
  walk, validation, and realization are generic.
- **Add a ruination area** â†’ `data/ruin_areas.py` (rooms) +
  `data/ruin_constants.py` (CHARACTER_AREAS / ROOM_REWARD / owners /
  shops). The planner picks it up from `RuinConfig`'s copies.
- **Change ruination behavior for a mode flag** â†’ adjust the table copies
  in `RuinConfig.__init__` (the `-maze` blocks are the template). Never
  mutate the `data/` tables.
- **Debug a bad seed** â†’ the planner runs offline:
  `RuinConfig(party, â€¦)` + `RuinPlanner(cfg, random.Random(seed))` +
  `grow()` + `finalize_plan()` in a REPL reproduces the exact plan with
  no ROM and no build.

## 8. Design properties worth knowing

- The cluster graph is always DAG-clean; shared-exit reattachment adds
  no duplicate pairs.
- Warp/town cooldown gating applies if *any* room of a candidate cluster
  is a warp/town room; the terminus is skipped as a target by cluster.
- Ruination accepts ~83% of growth attempts per try, so whole plans
  essentially always complete inside the 10-attempt retry loop and maps
  run lean (few rescue reserve pulls).

## 9. Event-layer mechanics

- **Derived predicates:** event files no longer keep flag or-chains;
  they declare their territory and call `self.doors_touched(<ROOM_SETS
  key>)` (or `rooms=(<room ids>,)` for areas without their own key).
  `doors/plan/modes.py door_rando_pool_keys` is the single key
  authority â€” `plan_mode` consumes it too, and
  `tests/doors/test_walk.py` pins the truth table. Ruination stays an
  explicit `or args.ruination_mode` where an event wants it (five
  events deliberately exclude it and handle ruination in their own
  hook).
- **Framework-dispatched hooks:** the Events loop fires any defined
  `door_rando_mod` / `dungeon_crawl_mod` / `ruination_mod` that `mod()`
  did not invoke inline (documented order: mod â†’ door_rando â†’ mode
  hook). Inline calls remain legal where variant code is interleaved
  mid-sequence; defining a hook and forgetting to wire it is no longer
  possible (`event/events.py _instrument_hooks/_dispatch_hooks`).
- **Plan query API:** `plan.destination_of / description_of /
  location_name`, and **`plan.gates`** â€” the unified gate table
  (exit â†’ key tuple, from the pool lock dicts; `character_gates()` for
  the local-character-gating subset). Listed in the spoiler's Door
  Rando section.
- **Manifest:** `python3 tools/mode_manifest.py [--markdown]` derives
  the mode Ã— event table (hooks / predicates / raw flags per file).

## 10. Regression protection

Any change that should not alter output is proven harmless by one run of
`python3 tools/golden_sweep.py -i <rom>` (15 mode x seed configs,
ROM + spoiler hashes vs the committed manifest). Deliberate output
changes re-record the manifest with `--update` and say why in the commit
message. CI runs the ROM-free planner suite on every push
(`tests/test_doors_v2.py`).
